# SQL 입력 명세 해석 메모

## 결론

- 이 요구사항은 `SQL 처리기가 커맨드라인을 통해 SQL 문장을 전달받을 수 있어야 한다`는 의미로 해석하는 것이 가장 자연스럽다.
- 가장 안전한 구현 방식은 `stdin` 기반 입력 처리이다.
- `./sql_processor < input.txt` 형태로 실행할 수 있다면 명세에 매우 잘 부합한다.
- `cat input.txt | ./sql_processor` 형태를 지원하는 것도 같은 취지에 맞는다.
- `./sql_processor input.txt`처럼 파일 경로를 CLI 인자로 받아 처리하는 방식도 충분히 타당하다.
- 반대로 프로그램 내부에서 `input.txt` 같은 고정 파일만 직접 열어 처리하는 방식은 명세 문구와는 거리가 있다.

## 근거

- 명세에는 `텍스트파일로 작성된 SQL문을 Command Line 명령을 통해 SQL처리기에 전달 할 수 있어야 합니다.`라고 적혀 있다.
- 여기서 핵심은 `프로그램 내부에서 파일을 읽는다`가 아니라 `커맨드라인 명령으로 SQL을 전달한다`는 점이다.
- 같은 문서에서 `CLI (Command Line Interface) 구현`도 별도로 강조하고 있다.

## 권장 설계

- 핵심 처리 함수는 `FILE *` 입력 스트림을 받아 SQL을 읽고 실행하도록 구성한다.
- 기본 입력 소스는 `stdin`으로 둔다.
- 필요하면 `argv[1]`이 있을 때 `fopen()`으로 파일을 열어 같은 로직으로 처리한다.
- 입력은 한 줄씩 읽더라도, 실제 실행은 `;`를 기준으로 SQL 문장을 모아서 수행하는 것이 좋다.
- 이렇게 설계하면 대화형 입력과 파일 리다이렉트 입력을 거의 동일한 코드로 처리할 수 있다.

## 권장 실행 예시

```bash
./sql_processor < input.txt
cat input.txt | ./sql_processor
./sql_processor input.txt
```

## 실제 구현 판단

- 우선 하나만 먼저 구현해야 한다면 `stdin` 입력 지원을 먼저 만드는 것이 가장 적절하다.
- 이 방식은 텍스트 파일 리다이렉트를 자연스럽게 지원하고, 명세에도 가장 직접적으로 부합한다.
- 파일 경로를 인자로 받는 기능은 이후 작은 확장으로 추가하면 된다.
