Week5 Mini React - Test Cases and Edge Cases

1. Node logic test cases

- useState keeps state between renders
- batched setState triggers only one extra update
- useEffect reruns only when deps change and runs cleanup
- useMemo reuses cached values when deps do not change
- hooks are blocked inside child components
- keyed diff detects insert, move, and text update
- empty stored task list survives reload
- corrupted JSON in localStorage falls back to seed tasks
- storage read failures fall back to seed tasks
- payloads without a tasks field fall back to seed tasks
- invalid stored task ids are repaired before adding new tasks
- duplicate stored task ids are deduplicated during hydration
- malformed task entries like null or numbers do not crash hydration
- blank stored task titles recover to default labels
- blank category and createdAt values recover to defaults
- object values in title/category/createdAt do not leak as "[object Object]"
- events for missing task targets do not dirty state
- edit draft changes do not recompute memoized task selectors until save
- storage write failures do not crash the app flow
- scheduled updates are ignored after destroy

2. Browser interaction test cases

- patch updates only the changed DOM nodes
- task creation prepends a new item
- task editing updates the title and delays memo recompute until save
- completion toggle updates state and styling
- completed filter shows only completed tasks
- search narrows the visible list
- useEffect persists tasks into localStorage
- memoized filtered list recomputes only when dependencies change

3. Edge cases explained in easy words

- If saved JSON is broken, the app should not die. It should start with default seed data.
- If localStorage read itself fails, the app should still open with default data.
- If the saved object does not even have a tasks field, the app should ignore it and recover.
- If a saved task id is not a real number, later add/edit/toggle flows can all break. The app now repairs that id.
- If two saved tasks share the same id, one click can affect both. The app now gives them unique ids again.
- If the saved tasks array contains null, numbers, or other bad shapes, hydration should not crash.
- If a saved title is blank, the UI should not show an empty item. It now falls back to Task n.
- If category or createdAt is blank, the screen should still look normal with defaults.
- If object values get into text fields, the UI should not show weird "[object Object]" text.
- If a delete/toggle event points at a task that does not exist, the app should ignore it cleanly.
- If persistence fails, users should still be able to keep using the in-memory UI.
- If a component is already destroyed, queued updates should not bring it back.
- Editing is split into two phases on purpose:
  draft typing changes only the edit buffer,
  saving changes the real tasks array.
  This lets the demo show when memoized calculations are reused and when they recompute.

4. Most critical case

- The most critical case was broken saved task ids.
- Easy version:
  one bad id could poison nextId,
  and then every new task after that could keep getting bad ids too.
- That means add, toggle, remove, keyed diff, and recent-task tracking could all become unreliable.
