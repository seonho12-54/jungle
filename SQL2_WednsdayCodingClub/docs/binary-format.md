# 바이너리 형식

이 문서는 현재 저장소 구현이 실제로 쓰는 바이너리 형식을 설명합니다.

## 공통 전제
- 정수 필드는 `uint16_t`, `uint32_t` 폭을 사용합니다.
- 멀티바이트 정수는 현재 little-endian 환경을 전제로 합니다.
- 헤더는 각 필드를 선언된 순서 그대로 연속 배치한 바이트열로 저장합니다.
- 현재 구현이 기대하는 헤더 크기는 `QSQL = 12 bytes`, `BKDB = 20 bytes`입니다.
- 다른 endianness 또는 다른 struct layout을 강제하는 환경은 지원 범위 밖입니다.

## Query 바이너리 형식: `QSQL`
하나의 SQL 배치를 통째로 저장합니다.

### 헤더
- magic: 4바이트 ASCII `QSQL`
- version: `uint32_t`, 현재 값 `1`
- sql_len: `uint32_t`, 뒤따르는 SQL payload 길이

### 페이로드
- 원본 SQL 배치 바이트열
- 문장 단위로 분리하지 않고 입력 문자열 그대로 저장

### 검증 규칙
- magic은 반드시 `QSQL`
- version은 반드시 `1`
- `sql_len`은 실제 payload 길이와 정확히 일치해야 함

## Data 바이너리 형식: `BKDB`
영속 저장용 `books` 테이블 파일입니다.

### 헤더
- magic: 4바이트 ASCII `BKDB`
- version: `uint32_t`, 현재 값 `1`
- rec_sz: `uint16_t`, 현재 값 `196`
- keep: `uint16_t`, 예약 필드, 현재 값 `0`
- cnt: `uint32_t`, 저장된 행 수
- next_id: `uint32_t`, 다음 자동 증가 id

### 레코드 하나의 구조
- `id`: `uint32_t`
- `title`: `char[96]`
- `author`: `char[64]`
- `genre`: `char[32]`

문자열 필드는 모두 고정 길이이며 남는 공간은 `0`으로 채웁니다.

### 검증 규칙
- magic은 반드시 `BKDB`
- version은 반드시 `1`
- rec_sz는 현재 구현이 기대하는 고정 레코드 크기와 같아야 함
- `sizeof(header) + cnt * rec_sz`가 실제 파일 크기와 정확히 같아야 함
- 헤더와 레코드 검증이 끝난 뒤 B+ 트리를 다시 구성함

## 캐시와 저장 동작
- 프로그램은 `data/books.bin`을 메모리 캐시로 읽어 옵니다.
- 모든 조회는 메모리 캐시를 기준으로 실행합니다.
- `WHERE id = n`과 `WHERE id BETWEEN a AND b`는 B+ 트리를 사용합니다.
- 그 외 조회는 선형 탐색을 사용합니다.
- 배치 출력은 전체 문장이 성공할 때까지 버퍼에만 쌓습니다.
- `INSERT`가 포함된 배치는 마지막까지 성공해야만 파일에 반영됩니다.
- 실패한 배치는 행 수와 `next_id`를 되돌리고 B+ 트리를 다시 만듭니다.
- 저장은 항상 `*.tmp` 파일을 먼저 쓴 뒤 원본과 교체합니다.

## 고정 길이 문자열을 쓰는 이유
- 파일 I/O가 단순합니다.
- 형식 검증이 쉽습니다.
- 테스트 데이터를 반복 생성하기 쉽습니다.
- 구조를 읽고 설명하기가 수월합니다.
